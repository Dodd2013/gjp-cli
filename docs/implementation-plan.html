📌 核心结论（先看这个）

通过对 passport.mygjp.com.cn 前端 JS 的完整逆向，登录流程可拆解为两块：可完全复现的加密 与 需要引导获取的指纹 Token。

✅ 已验证可行（纯 TS，无浏览器）
  • RSA-1024 加密：公钥 e=0x10001、模数 n 硬编码在前端，已移植为原生 BigInt 实现。Bun 实跑产出 256 hex / 128 字节密文，与 HAR 抓包格式完全一致，随机填充（PKCS#1 type2）也正常。
  • 登录请求：纯 fetch POST，JSON body 结构已完全掌握。
  • 会话维持：登录成功后服务端下发 HttpOnly Cookie（HAR 中被 Chrome 剥离），用 Cookie Jar 持久化即可，所有 /jxc/ 接口靠它鉴权。
⚠️ 唯一风险点：设备指纹 Token（deviceId / ati / pati）

登录请求里这三个字段来自 京东 eid、拼多多 pati、安全 SDK _ati —— 它们本质是反爬虫指纹，纯算法生成极其困难。是否严格校验取决于服务端（无法离线判定），方案设计了 三级降级策略应对，详见第 4 节。

1.4 填充算法（Ohdave RSA.js 自定义 PKCS#1 type2 变体）

这是非标准填充（明文字节逆序 + 自定义标记位），所以不能直接用 Node crypto.publicEncrypt。必须手动构造数据块：

每个 128 字节数据块布局（低位→高位）：
┌─────────────────────────────────────────────────────────────┐
│ 明文字节（逆序） │ 0x00 │ 随机字节(1..254) × N │ 0x02 │ 0x00 │
└─────────────────────────────────────────────────────────────┘
   → 打包成 BigInt → 计算 block^65537 mod n → 输出 256 hex

// 关键特性：随机字节使"同一明文每次密文不同"

🔑 第 2 节 · 完整鉴权流程

登录后鉴权靠 HttpOnly Cookie（HAR 中被 Chrome 出于安全剥离，但抓包显示 afterLogin 等请求无任何显式 token 仍返回 200，证明 cookie 在 cookie jar 里自动携带）。

(可选) POST /errorcount/get · 查询错误次数
body: {productId:88, companyName, userName: RSA(用户名)}。若 errorCount≥3 触发滑块验证码。正常登录无需。
POST passport.mygjp.com.cn/api/ngpLogin · 核心登录
body: {userName, password, companyName, deviceId, ati, pati, validateCode, validateId, https:true, loginType}。返回 redirectKey + 设置 passport 域 cookie。
GET ngpkj.wsgjp.com.cn/main.html?redirectKey=…×tamp=…&sign=… · 302 换 session
服务端校验 redirectKey+sign，下发 ngpkj 域 HttpOnly 会话 Cookie，302 跳到 main.html?v=…。
POST ngpkj.wsgjp.com.cn/jxc/recordsheet/sys/afterLogin · 会话定型
空 body。返回 {"code":"200","message":"操作成功"}。此后所有 /jxc/ 接口可用。
Cookie Jar 持久化 → 后续业务调用
把会话 cookie 写入本地加密文件（~/.gjp/session.json），复用直至过期，过期自动重登。

🛡️ 第 4 节 · 指纹 Token 三级降级策略

这是整个方案唯一的不确定项。三个指纹字段由反爬 SDK 生成，纯 TS 无法凭空伪造合法值。策略：从简单到复杂依次尝试，命中即停。

① 空值试探

先发空/占位值，很多系统仅在异常时才严格校验指纹。

  • deviceId/ati/pati 全留空
  • 凭据正确时大概率放行
  • 零成本，先试这个

预期：60% 系统能过

② 一次性引导（推荐）

用户在自己浏览器 手动（非自动化）取一次值，本地永久保存。指纹通常按天/周有效。

  • CLI 引导命令 gjp auth bootstrap
  • 打印提取步骤（DevTools 复制）
  • 存入本地配置，自动续期检测

这不是 Playwright，是 30 秒手动操作

③ SDK 最小化复现

若 ② 过期太快（几小时），则拉取 SDK 源码，用 Bun 跑 JS 沙箱生成。

  • 分析 sdk-glue.js / bdms.js
  • 无 DOM 依赖部分用 Bun 执行
  • 成本最高，最后手段

保底方案

💡 关键判断：先用真实账号跑一次 Tier ①。若登录返回 200 且拿到会话 cookie → 整个指纹问题不存在，方案 100% 纯 HTTP。若被拒 → 升级 Tier ②。这个判断在拿到测试账号后 10 分钟内就能完成。
